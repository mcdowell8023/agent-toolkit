# Init Deep Fallback

Cross-platform fallback for generating **hierarchical AGENTS.md** documentation when no platform-native or plugin-provided init-deep tool is available.

## Why this exists

| Tool | Available in |
|------|------|
| OMC `deepinit` (Yeachan Heo) | Claude Code + OMC plugin |
| OMO `/init-deep` (code-yeongyu) | OpenCode + OMO package |
| Claude Code `/init` (builtin) | Single CLAUDE.md only — not hierarchical |
| OpenCode `/init` (builtin) | Single AGENTS.md only — not hierarchical |
| Codex | No init / init-deep / hierarchy support |
| **This skill** | **Any platform, no plugin required** |

agent-gates ships this skill to give all users a baseline hierarchy capability without forcing OMC/OMO plugin installation.

> **Future**: When users adopt Rampart (next-generation base layer), `rampart init-deep` will auto-route to OMC / OMO / this fallback based on environment. This skill becomes the cross-platform implementation Rampart calls. Direct use as a skill remains supported.

## Workflow

### Step 1: Scan project directory tree

Use platform-appropriate tools (Glob, Bash `find`) to enumerate directories at depth ≤ 3. Filter:

**Exclude** (never generate AGENTS.md for):
- `.git`, `node_modules`, `dist`, `build`, `target`, `out`, `coverage`
- `.next`, `.nuxt`, `.cache`, `.parcel-cache`, `.vite`
- `__pycache__`, `.venv`, `venv`, `.tox`, `.mypy_cache`, `.pytest_cache`
- `vendor`, `bower_components`
- `.agent/memory`, `.tmp`, `tmp`, `logs`
- Hidden dirs starting with `.` unless explicitly meaningful (e.g. `.github` is OK)

**Include** (generate AGENTS.md for):
- Top-level functional directories: `src`, `app`, `lib`, `packages`, `apps`, `components`, `services`, `api`, `tests`, `e2e`, `docs`, `scripts`, `hooks`, `skills`
- Any direct subdir of project root containing source code files (`.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.go`, `.rs`, `.java`, `.kt`, `.rb`, `.php`, `.cs`)
- Custom matchers based on project type (e.g., `infrastructure/`, `terraform/`, `k8s/`, `helm/` for IaC projects)

### Step 2: Generate root AGENTS.md

```markdown
# AGENTS.md — {{PROJECT_NAME}}

> AI-readable project documentation. Generated by init-deep-fallback on {{DATE}}.

## What is {{PROJECT_NAME}}

{{ONE_LINE_DESCRIPTION}}  (agent fills based on README / package.json / inspection)

## Directory Structure

{{TREE_OVERVIEW}}  (agent fills: top-level dirs + one-line purpose each)

## Key Workflows

- Read `.agent/PROGRESS.md` first in new sessions
- See `<subdir>/AGENTS.md` for per-directory context (hierarchical)
- TDD / cross-review evidence required per `agent-workflow-rules` SKILL

## Subdirectory References

{{SUBDIR_LIST}}  (agent fills, e.g. `- [src/](src/AGENTS.md) — Source code`)
```

### Step 3: Generate per-subdirectory AGENTS.md

For each meaningful subdirectory from Step 1, generate:

```markdown
<!-- Parent: ../AGENTS.md -->

# {{SUBDIR_PATH}} — AGENTS.md

> See parent: [`../AGENTS.md`](../AGENTS.md)

## Purpose

{{PURPOSE}}  (agent infers from file content / naming)

## Key Files

{{FILE_LIST}}  (agent picks 3-5 most important files in this dir)

## Conventions

{{CONVENTIONS}}  (agent inspects: test naming, import patterns, file naming)

## Subdirectories

{{NESTED_SUBDIRS}}  (recursive; depth ≤ 3 total from root)
```

### Step 4: Validate hierarchy integrity

- All non-root AGENTS.md files contain `<!-- Parent: ../AGENTS.md -->` marker
- All root + child links resolve (no broken file refs)
- No file > 200 lines (each AGENTS.md is concise, point of contact)

### Step 5: Idempotency

If AGENTS.md already exists:
- Preserve sections marked `<!-- MANUAL -->` ... `<!-- /MANUAL -->`
- Refresh other sections based on current project state
- Never overwrite without confirmation

## Output Format

For each generated AGENTS.md, output one line:
```
✓ Created/Updated: <path> (N lines, parent: <parent>)
```

Final summary:
```
Generated N AGENTS.md files across M directories.
Root: ./AGENTS.md
Children: src/, lib/, tests/, ... (depth 1-3)
```

## What This Skill Does NOT Do

- **Deep semantic analysis** — agent fills `{{PURPOSE}}` / `{{CONVENTIONS}}` based on file inspection, not full understanding. Users should review and refine.
- **LLM-driven content for every line** — this is a structural scaffold; agent fills based on observable file content + naming. Heavy semantic content is the user's job.
- **Match OMC deepinit feature parity** — OMC deepinit is more thorough (custom tagging, conflict resolution, advanced templates). This is a baseline fallback.

## Relationship

- **agent-gates `init-project-gates` Step 6** uses this skill as the cross-platform fallback when OMC deepinit / OMO /init-deep are unavailable
- **Rampart (future)** will provide `rampart init-deep` command that auto-detects and routes — this skill becomes its cross-platform implementation
- **OMC deepinit / OMO /init-deep**: prefer these when available (more features)
